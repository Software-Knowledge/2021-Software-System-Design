# 1. 课程背景

## 1.1. 为什么要研究软件设计体系结构？
1. 软件(IT)系统无处不在
2. 每个软件密集型系统都有一个软件设计和体系结构
3. 软件设计和体系结构是实践，教育和研究中越来越重要的领域
4. 专业：软件架构师
5. 作为研究领域
   1. 最初于1960年左右开始
   2. 自1990年以来备受关注
6. 本课程是关于
   1. 软件设计和体系结构的概念，原理，方法和模式
   2. 软件设计和体系结构的最新实践


## 1.2. 将要学习到的东西
1. 理解软件设计和体系结构的概念和原理
2. 通过考虑需求或通过反向体系结构来创建软件体系结构
3. 在创建软件体系结构和设计的时候应用设计模式、风格、中间件技术和框架
4. 分析软件设计和评估软件体系结构的系统性
5. 理解艺术式的设计是如何应用到软件设计与体系结构中的
6. 理解软件设计与软件体系结构之间的关系，以及其他软件工程的领域话题

## 2.2. 什么是软件体系结构
1. 定义1：程序或计算系统的软件体系结构是系统的一个或多个结构，其中包括软件组件，这些组件的外部可见属性以及它们之间的关系。 [软件工程学院(SEI)]
2. 定义2：系统的基本组织，体现在其组件，它们之间的相互关系以及环境以及支配其设计和演进的原则。[IEEE 1471 -2000 有关软件密集型系统的体系结构描述的推荐做法]

## 2.3. 体系结构 vs 设计
1. 它是关于软件设计
   1. 所有的体系结构都是软件设计，但不是所有的软件设计都是体系结构。
   2. 体系结构是设计过程的一个过程
2. 其他观点
   1. 更高层的设计
   2. 设计决策的组合
   3. 地区
3. 系统的结构或组织
   1. 元素：部件和连接件
   2. 关系：静态和动态的关系
4. 属性：元素，元素组和整个系统

## 2.4. 体系结构 vs 结构
1. 将系统分解成部件/模块/子系统
2. 体系结构定义：
   1. 部件接口：部件可以做什么？
   2. 部件交流和依赖：部件可以怎么沟通交流？
   3. 部件职责：当我们询问它时，部件需要精确的知道自己将要做什么？

## 2.6. 体系结构指定通信
1. 通信需要：
   1. **数据**通过机器传递，比如
      1. 函数调用
      2. 远程方法调用
      3. 异步信息
   2. **控制**流
      1. 组件间的信息流来满足需要的功能
      2. 序列化的
      3. 并发/并行
      4. 同步

## 2.7. 体系结构强调非功能性需求(NFA)
1. **非功能性需求(Non-functional requirements)**定义了**系统运行的有多好**
2. 非功能性需求很少在功能性性需求中很少被发现
   1. 又名体系结构需求
   2. 必须通过体系结构引出
3. 非功能性需求
   1. 技术约束
   2. 商业约束
   3. 质量属性
4. 讨论：质量属性列表

## 2.8. 设计是一种抽象
1. 体系结构提供了设计的更高层抽象视角
   1. 隐藏设计的复杂性和实现
   2. 可能是或者可能不是体系结构元素和软件元素之间的直接映射
2. 黑盒设计 和 白盒设计
   1. 是对系统结构和交互的非正式描述
   2. 描述在体系结构中内嵌的设计哲学
3. 讨论：为什么在设计中使用抽象？

## 2.9. 体系结构视角
1. 软件体系结构表示了一个复杂的设计制品
2. 很多体系结构的可能视图：类比建筑-平面图，外部设计，电力设计，水暖，空气调节

## 3.1. K.Kruchen的4+1视图模型
1. 逻辑视图：描述了体系结构中在体系结构上明显重要的元素以及他们之间的关系
2. 过程视图：描述了体系结构中的并发和交流元素
3. 物理视图：描述了主要过程和部件是如何映射到应用硬件上的
4. 发展视图：描述了软件部件是如何在软件内部组织的，比如配置管理工具
5. 使用体系结构的情况：描述了体系结构的需求，关系到了超过一个常规的视图

## 3.2. 体系结构和软件体系结构
> 体系结构设计了满足人类需求的结构

1. 体系结构的作用仍然保持着一致
   1. 监听客户端，理解整体的需求
   2. 仔细检查灵活性
   3. 形成一个实际的结构版本，创建一个蓝图
   4. 监督构建过程，保证是符合规范的
   5. 引导在暴风雨式的设计变更、危机和歧义性中的版本
2. 软件体系结构则是监督软件的构建过程：开发人员、工程师和设计者
3. 有名的软件体系结构
   1. Bill Gates: Chief Software Architect of Microsoft
   2. Tim Bernrs-Lee: Inventor and Chief Architect of World Wide Web
   3. Roy Fielding: Representational State Transfer (REST)

## 3.3. 软件体系结构在做什么

3. 技术知识：深入理解技术领域
4. 风险管理：
   1. 与设计、技术决策相关的风险

## 3.9. 软件体系结构和体系结构知识领域
1. 软件设计基本原理
   1. 整体设计原理
   2. 上下文：软件发展生命周期-需求、设计、编码和测试
   3. 设计过程：决策、活动、可运行产品
2. 核心问题(技术)：一致性、事件控制和处理、分发、异常处理、交互系统、持久化
3. 软件结构和体系结构
   1. 体系结构结构和视点
   2. 体系结构风格和模式(宏观体系结构)
   3. 设计模式(微观体系结构)
4. 软件设计方法
   1. 体系结构方法，比如属性驱动的设计
   2. 设计方法，比如动态系统发展方法
5. 软件设计的质量分析和评估
   1. 质量属性
   2. 质量分析和评估方法、技术和工具
      1. 设计回顾：比如SEI的体系结构权衡分析方法
      2. 静态分析和动态分析
      3. 模拟和原型
   3. 度量
      1. 矩阵：体系结构级别
      2. 技术特有度量指标
6. 设计建模和展示
   1. 体系结构和设计符号(体系结构描述语言，Architecture Description Languages，ADL)
   2. UML
   3. 设计文档(意见或其他)
   4. 其他：在活动、关注点和领域上的不同，比如ACME，Rapide
